Property Taxonomy v3

Property Types

A canonical property classification system that every RentSync platform translates to and from. Store canonical, translate for display. One source of truth for residential, commercial, and mixed-use properties across all platforms.

The model

Every listing is classified using a small set of fields. Only buildingSubtype and unit are truly required — everything else is derived or optional.

Two levels

Data exists at two levels. Property-level fields can be derived from units or input manually — the system fills what it can, but you can always override. Unit-level fields are what landlords provide for each listing.

Property level

Can be derived from units, or input manually:

sector[] — rolled up from unit sectors
category[] — rolled up from unit categories res
single_multi — from buildingType, or set directly res
community — optional wrapper name res
buildingSummary[] — all building forms present
unitSummary[] — all unit classifications
buildingAmenities{} — building-level amenities

Unit level

Set per unit by the landlord:

buildingType · buildingSubtype — the structure
unit · unitType — what's being listed
ownership · transaction — the deal
lease_type · term · deposit — lease details
condo_fees — when ownership = condo
category — market segment res
beds · baths res · sqft · price · price_per — details
unitAmenities{} — unit amenities

Other cascade

Land, leisure, and mixed-use. Each has its own natural unit vocabulary — lots and parcels for land, slips and stalls for leisure.

Land

Subtypes: agricultural, vacant, acreage, subdivision, industrial_land, commercial_land, residential_land

Leisure

Subtypes: marina, campground, rv_park, stable, golf_course, ski_resort, vineyard

Units: entire · slip · berth · dock · site · pad · stall · paddock

The deal

Deal fields describe the transaction. All are per-unit — different units in the same building can have different ownership, transaction types, and terms.

ValueWhat it meansTypical sector
standardStandard residential leaseResidential

▶ term How long? · Optional · free-form duration

Free-form: any number + period. No preset values — landlord sets what they want.

"term": { "amount": 90, "period": "days" }     ← short-term / vacation
"term": { "amount": 1, "period": "weeks" }     ← weekly rental

daysVacation, short-term stays
weeksFurnished rentals, sublets
monthsStandard residential leases
yearsCommercial leases

Free-form amount. No preset structure — varies by province and sector.

Categories

Category is per-unit (residential only). Different units in the same building can have different categories. Property-level category is rolled up from units.

conventional senior student military corporate vacation short_term co_living affordable luxury

Categories drive amenity shortlists — when a landlord sets category to "senior," the listing form surfaces accessibility, medical, and safety amenities first. See the amenities spreadsheet for the full shortlist matrix.

The system never rejects data. If someone sends single_multi: "multi_family" for a house, we store it. The UI can flag it as unusual, but the API accepts it. This is the permissive model.

Design decisions

Condo is ownership, not building type

A condo unit can exist in a high-rise, townhouse, or house. "Condo" moved from Property Type to the ownership field. On ILS, "Condo" still appears as a search filter — handled by the translation layer.

Community is a wrapper, not a type

A community groups multiple buildings under a brand name for display. Each building within is classified independently. The community bumps the building down one level in the ILS display hierarchy.

Derived over declared

SF/MF is derived from buildingType, not self-declared. Sector is derived from units. This eliminates billing distortion — nobody can game their classification for a better rate.

Unit-level buildingType override

Units can declare their own buildingType when the physical space differs from the property (e.g., podium townhouses in a tower, laneway home behind a house). Null = inherit from property.

Mixed-use as its own buildingType

mixed_use_bldg exists in the Other cascade with subtypes like live_work and retail_residential. Both residential and commercial data apply. Sector = "mixed_use, residential, commercial."

Billing decoupled from classification

"Nobody should be changing how they structure their data to save on billing." (Steven Rai). The data model reflects reality, not billing optimization.

Key rules

1

Store canonical, translate for display. Every platform maps to/from the canonical model.

2

Derived over declared. Fields that can be computed (sector, single_multi) should not be user-declared.

3

Condo is ownership, not structure. Same applies to co-op.

4

Community is a wrapper, not a type. For display grouping only.

5

Billing decoupled from classification. No financial incentive should influence data accuracy.

6

Permissive model. The system never rejects data. Canonical values are what we recognize for search/filter, not what we enforce.

7

Unit-level overrides. Units can declare their own buildingType, sector, and category when they differ from the property.

8

Only buildingSubtype and unit are required. Everything else is derived, optional, or has a sensible default.
