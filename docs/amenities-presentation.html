The data model

Amenities use BSTK-native data types — Bool, Enum, Bool+String — with one addition: Bool + detail, which lets a simple boolean grow subtypes over time without migration.

Bool

228 amenities — BSTK native

Simple presence. Dishwasher, elevator, balcony.

"dishwasher": true
Enum

~70 amenities — BSTK native

Pick one. Flooring, fireplace, furnishings, utilities.

"flooring": "hardwood"
"electricity": "included"
Bool + detail

~15 amenities — new pattern

Bool for search, detail array for specifics.

"pool": true
"pool_detail": ["heated", "outdoor"]
Bool + String

2 amenities — BSTK native

"Other" fields with free text.

"parking_other": true
"parking_other_text": "Tandem"

The Bool + detail pattern

This is the key addition to BSTK's existing types. It separates two questions that are genuinely different:

Bool     →  "do you have it?"   →  the search filter   →  never changes
Detail   →  "what kind?"        →  the listing page    →  grows over time

When a Bool amenity needs subtypes later, you add the _detail companion. The Bool never changes. No migration. Old data stays valid forever.

Today:      "freezer": true
Next year:  "freezer": true                          ← unchanged
            "freezer_detail": ["upright"]             ← new companion
Old data:   "freezer": true (no detail)              ← still valid

Data types and storage

BSTK-compatible. Bool is true, Enum is a string, Bool+detail is true plus a string[] companion.

Bool:          "dishwasher":    true

Building vs unit level

Some amenities exist at both levels. They answer different questions:

Building level

What's available on site. The policy.

Unit level

What's included with this unit. Or an override.

Parking

Building — what's on site:
"parking":      ["available"]
"parking_type": ["underground", "visitor", "ev_charging"]

Unit — what comes with this lease:
"parking":      ["included_in_rent"]
"parking_type": ["underground"]       ← 1 underground spot
"parking_qty":  ["1"]

Pet policy

Building — the policy:
"pet_policy":     ["allowed"]
"pet_type":       ["cat", "dog"]
"pet_size_limit": ["medium_50lbs"]

Unit A (ground floor, near garden):
"pet_policy":     ["not_allowed"]   ← override

Unit B (upper floor):
← key absent = inherits building policy (allowed, cats + dogs, medium max)

The inheritance rule

Category at both levels

category exists at both the property and unit level. The property-level category is rolled up from units, but individual units can have different categories — a building might have both conventional and senior units.

Property level (calculated — rolled up from units):
"category": ["conventional", "senior"]    ← building has both types

Unit level (input — set per unit):
Unit A: "category": "senior"              ← assisted living unit
Unit B: "category": "conventional"        ← standard rental unit

The category shortlist applies per unit — when the landlord sets a unit's category to "senior," that unit's amenity form shows the senior shortlist. Other units in the same building can use different shortlists.

Key present in unitAmenities → unit value wins (override).
Key absent from unitAmenities → inherits from buildingAmenities.
Key absent from both → not specified.

Category shortlists

When a landlord selects a category, the form shortlists relevant amenities. This doesn't restrict what's available — it surfaces the most relevant ones first, organized by group.

Migration-free growth

Because Bool+detail separates presence from specifics, amenities evolve without touching existing data.

Bool → Bool + detail (add subtypes)

Today:      "freezer": true                         ← Bool, no detail
Next year:  "freezer": true                         ← Bool unchanged
            "freezer_detail": ["upright"]            ← new companion field
Old data:   "freezer": true (no detail)              ← still valid forever

Search: "has a freezer" → freezer: true (matches all)
Search: "upright freezer" → freezer_detail contains "upright"

New value added to existing detail

Today:  pool_detail values: [heated, indoor, outdoor, hot_tub, jacuzzi]
Later:  pool_detail values: [heated, indoor, outdoor, hot_tub, jacuzzi, saltwater, infinity]
Old data untouched. New values just appear in the dropdown.

New Enum value added

Today:  electricity: [included, tenant_pays, not_available]
Later:  electricity: [included, tenant_pays, partially_included, seasonal, not_available]
Just append to the values list. Old data stays valid.

Custom → promoted to detail (the pipeline)

Day 1:    custom: [{ parent: "pool", text: "Saltwater pool" }]
Day 200:  "saltwater" added to pool_detail canonical values
          custom entries matching "Saltwater pool" → remapped to pool_detail: ["saltwater"]
          Bool "pool: true" never touched. No migration.

Confidence pipeline

AI detections are suggestions, not auto-populated values. The landlord sees a pre-filled form with AI detections as checkboxes — high confidence items are pre-checked, medium confidence items are highlighted but unchecked.

Step 1 — AI runs on uploaded photos:
  suggested: { dishwasher: high, pool: high, pool_detail: ["outdoor"] medium, high_ceilings: medium }

Step 2 — Landlord confirms on the form:
  ☑ Dishwasher (detected in photo 3)
  ☑ Pool
      ☑ Outdoor (detected)     ☐ Heated (landlord adds)
  ☐ High ceilings (confirm?)

Step 3 — Stored after confirmation:
  "dishwasher":    true
  "pool":          true
  "pool_detail":   ["outdoor", "heated"]   ← AI + landlord combined

AI feeds the custom promotion pipeline

When AI detects something with no canonical match, it goes to the custom array tagged as ai_detected. This makes AI a discovery engine — the system learns what amenities exist in the real world, and frequently-detected items become candidates for canonical promotion.

"custom": [
  { "parent": "pool", "text": "Infinity edge pool", "source": "ai_detected" },
  { "parent": null,   "text": "Rooftop solar panels", "source": "ai_detected" }
]

When "infinity edge" appears across 500+ properties → promote to pool_detail
Three data sources, one model: landlord input (manual, high confidence), AI detection (automated, confirmed by landlord), and custom free text (not searchable until promoted). All feed the same Bool / Enum / detail fields.

Handling existing custom amenities

Properties already in the system may have free-text custom amenities. Here's how they migrate into the new model:

Step 1 — Scan existing customs against canonical list:
  "Stainless steel appliances"  →  matches stainless_steel_appliances: true
  "In-unit washer/dryer"        →  matches laundry: true + detail: ["in_suite","washer","dryer"]
  "Underground parking"         →  matches parking_type: true + detail: ["underground"]

Step 2 — Fuzzy matches flagged for review:
  "W/D in unit"                 →  likely laundry (in_suite) — confirm
  "AC"                          →  likely ac_type — confirm
  "Renovated"                   →  likely renovation: "recently_renovated" — confirm

Step 3 — No match = stays as custom, tagged to nearest parent:
  "Tesla wall charger"          →  custom, parent: "parking_type"
  "Smart home system"           →  custom, parent: null
  "Japanese soaking tub"        →  custom, parent: null
Existing customs don't disappear. They either map to canonical values (auto or confirmed), or stay as customs tagged to a parent. The migration is additive — canonical fields get populated, matched customs get archived with a "migrated_to" reference, unmatched customs persist as-is. AI detection can also help: run the tagger on existing photos to auto-suggest canonical values for listings that only had free-text customs.

The rules

1

Bool is bool. Enum is enum. BSTK-native types. No unnecessary abstraction.

2

When a Bool needs subtypes, add a _detail companion. Bool never changes. No migration.

3

_detail is always string[] — array of canonical values from the values list.

4

Utilities are Enum amenities with values [included, tenant_pays, not_available]. Same shape as flooring.

5

AI detections map to the same fields. Bool for presence, _detail for specifics, custom for unknowns.

6

Dual-level amenities use inheritance. Unit overrides building. Absent = inherit.

7

Custom entries are never searchable until promoted to canonical.

8

Category ✓ marks mean "shortlist on the form", not "available for this category."
