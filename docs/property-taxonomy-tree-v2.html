Platform Display Concepts — Not Structure Types

ILS sites like Rentals.ca show "Room" and "Condo" as top-level Property Types in their UI. But in our canonical model, these aren't Structure Types — they're resolved through other dimensions and the platform mapping layer. This is how we support third-party feeds (Roomies, MLS) without needing separate structure types.

🛏️ "Room" on ILS = Unit Type, not Structure Type

A room exists inside a structure (house, apartment, townhouse). The room is the unit type, not the building type.

// Mike lists a room in his townhouse
structure_type: multi_unit_lr
subtype: townhouse
unit_type: room_private ← key field
purpose: student
ILS mapping rule:
If unit_type = room_* → display as Property Type: "Room" on Rentals.ca, RentBoard, RentCanada.
If unit_type = room_* → display as "Room For Rent" on RentFaster.
🏙️ "Condo" on ILS = Ownership Attribute, not Structure Type

"Condo" describes who owns the unit, not what the building looks like. A high-rise, townhouse, or house can all be condos.

// MLS sends a "Condo" listing
structure_type: apartment_bldg
subtype: high_rise
unit_type: entire
ownership: condo ← key field
ILS mapping rule:
If ownership = condo → display as Property Type: "Condo" on Rentals.ca.
Search for "Condo" → queries all listings where ownership = condo, regardless of structure type.
📥 Inbound: Sparse Third-Party Data (Roomies, MLS)
Roomies sends
type: "Room"
city: "Toronto"
price: $800
// no structure info
// no ownership info
→
We store (canonical)
sector: residential
structure_type: res_other ← unknown
subtype: null
unit_type: room_private
listing_type: rent
purpose: standard ← default
source: roomies
needs_enrichment: true
→
Displays on ILS
// Rentals.ca:
Property Type: "Room" ✓

// RentFaster:
SU Type: "Room For Rent" ✓

// Structure unknown but
// listing still works ✓
Key: The listing displays correctly because ILS outbound mapping only needs unit_type to resolve "Room." Structure type is unknown but doesn't block display. The needs_enrichment flag lets us backfill later.
Required vs Optional — By Data Source

Not every source can provide every dimension. The model must work with partial data from external feeds while requiring full granularity from internal platforms. Catch-all "Other" values are load-bearing for this — they're where sparse inbound data lands.

Dimension BSTK / RentSync
Enterprise 		ILS Direct
Landlord form 		External Feed
Roomies, MLS, Zumper 		Default if missing
Sector REQUIRED REQUIRED (inferred) REQUIRED (inferred) Infer from context — almost always determinable
Structure Type REQUIRED REQUIRED (asked) Optional res_other or com_other
Structure Subtype Optional Optional Optional null — infer later or leave blank
Unit Type REQUIRED REQUIRED (asked) REQUIRED (usually provided) entire — safest assumption
Listing Type REQUIRED REQUIRED (asked) REQUIRED (usually provided) rent — most common default
Purpose Optional Optional Optional standard
Ownership Optional Optional Optional (infer if possible) null — unknown is fine
Hard minimum for any listing: Sector + Unit Type + Listing Type. If we have these three, the listing can display on ILS sites even with everything else unknown.
"Other" catch-alls are load-bearing. They're not just nice-to-haves — they're where sparse inbound data from Roomies, MLS, and future integrations lands until it can be enriched. Without them, we'd either reject data or force-fit it.
