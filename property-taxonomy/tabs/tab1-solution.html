What We're Solving

47+ inconsistent property types across BSTK, RentSync, ILS, and Spacelist. "Apartment" means different things on different platforms. Rooms and basements get forced into building-level classifications. There's no clean way to represent loft vs studio vs garden suite.

The solution: one canonical data model with 6 orthogonal dimensions plus a platform mapping layer. Store once, display everywhere.

The Six Dimensions
Think of them as questions we answer about each listing.
Dimension Question It Answers Example Values Required?
Sector What broad category? residential, commercial, recreation, other Required
Property Type What kind of property? house, townhouse_plex, apartment_bldg, office, retail Optional*
Subtype What specific variation? detached, mobile, cabin, low_rise, duplex, class_a Optional
Variant What style or refinement? loft, studio, penthouse, bachelor, bungalow Optional
Unit Type What are you listing? apartment, basement, room, entire, suite, spot Required
Transaction What's the deal? rent, sale, lease Required
* Property Type is optional for "unit-first" listings. Some listings are defined by the unit, not the building. When someone lists "an apartment" or "a room for rent," they don't always know or care whether it's in a low-rise or high-rise. For these unit-first listings (apartment, basement, room, office, parking), Property Type is nullable — the unit type IS the classification. When the building info is known, it's stored as enrichment.

In practice, natural language collapses the redundant parts. The system stores the full canonical data; humans only say what's meaningful.

Collapse Patterns
How canonical data becomes natural speech
Scenario What Collapses Sentence Shape Example
Unit-first, simple Property Type, Subtype, Variant hidden [unit type] for [transaction] "Apartment for rent"
Unit-first + variant Property Type, Subtype hidden [variant] [unit type] for [transaction] "Loft apartment for rent"
Structure-first, entire Unit Type ("entire") hidden [subtype] or [variant] for [transaction] "Townhouse for rent" / "Bungalow for sale"
Structure-first, portion Preposition joins unit to structure [unit type] in/of/at [subtype] for [transaction] "Suite in triplex for rent"

Two Paths to a Listing

Unit-First Property Type optional

"I'm listing an apartment" — the unit IS the product. Building details are background context.

Apartment Basement Room Office Parking

Example: "2BR apartment for rent" — renter doesn't care if it's low-rise or mid-rise

Structure-First Property Type required

"I'm listing my house" — the building IS the product. Unit type tells you what portion.

Entire Main Floor Suite Floor Spot

Example: "3BR detached house for rent" — the house IS the listing

Same model, same database — just two entry points. Both paths store data in the same 6 dimensions. The only difference is whether Property Type is required (structure-first) or nullable (unit-first). When a unit-first listing does have building info — "2BR apartment in a mid-rise" — we store it as enrichment. Nothing is lost.

Platform mappings are configuration, not code. Each platform has a mapping table that translates canonical values to its display vocabulary. When we acquire a new platform, we add a mapping table — we don't restructure the data model.

Cardinality & Uniqueness

Dimension Cardinality Unique to Parent? Notes
Sector 1:1 — (top level)
Property Type 1:1 (nullable) Yes Each type belongs to exactly one sector
Subtype 1:1 Yes Each subtype belongs to exactly one type
Variant 1:1 (nullable) No Variants can appear under multiple subtypes (e.g., Bachelor under Low/Mid/High Rise, Walk-up)
Unit Type 1:1 No Unit types are valid across multiple property types
Transaction 1:1 No rent/sale/lease valid across property types
What the uniqueness break at Variant means in practice: when a renter searches for "bachelor," the system returns results across all Apt Bldg subtypes because Bachelor isn't unique to one subtype. This is intentional — variants describe style, and the same style can occur across different building sizes.

Backwards Compatibility

1:1
Each old type has one canonical mapping
0
Data loss during migration
Other
Catch-all at every level for unrecognized values
Migration is deterministic. Every existing property type (from BSTK, RentSync, ILS, Spacelist) has exactly one canonical mapping. "Other" values at every level catch anything that doesn't map cleanly — this also future-proofs for new property types from acquisitions. Full migration table available in the reference spreadsheet.
